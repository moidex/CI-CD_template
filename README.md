# CI/CD_template

Итак, вы решили задеплоить свой первый проект...

Мы будем делать это на примере Python 3.12 и простейшего Flask-приложения. Деплоить будем на **Yandex Cloud** и **Serverless Containers**.

## Начало работы

У нас есть почти готовый `.yml` файл для пайплайна. Он лежит в папке `.github/workflows`.

В нём есть комментарии: одни понятны сразу, другие потребуют дополнительных действий. В этом README описаны подробные процессы для настройки нужных параметров.

> **Важно:** Все места в `.yml`, где нужно заглянуть в этот файл, помечены так: `[[README_тема_которую_надо_смотреть]]`.  
> После изучения раздела замените квадратные скобки на полученное значение.

## Секреты GitHub

Секреты GitHub — это секреты, которые используются при выполнении пайплайна. Они хранятся внутри репозитория. Для каждого репозитория секреты свои.

**Как создать секрет:**

1. Перейдите в настройки репозитория.
2. Откройте вкладку **Secrets and variables** → **Actions**.
3. Создайте секрет в формате «ключ — значение».

## Получение JSON-авторизации для сервисного аккаунта

Чтобы создать сервисный аккаунт, нужно добавить новый сервис в облако. Сокращённо он называется **IAM** (полное название — на скриншоте).

![Создание IAM](images/IAM/1.png)

**Шаг 1.** Создаём сервисный аккаунт. Мы рассмотрим самый простой путь, но он не является самым безопасным.

Вводим название и выдаём права. Можно выдать права к конкретным сервисам, но мы дадим аккаунту права администратора, чтобы избежать лишних проблем.

![Создание сервисного аккаунта](images/IAM/2.png)

**Шаг 2.** Заходим внутрь созданного аккаунта и создаём ключ доступа. Нажимаем кнопки, как на скриншоте.

![Создание ключа для сервисного аккаунта](images/IAM/3.png)

Во всплывшем окне оставляем всё как есть. Можно ввести описание ключа, чтобы потом понимать, какой ключ где используется.

Появится окно с ключами. **Важно:** скачайте их — посмотреть повторно не получится. Если потеряете, придётся пересоздавать.

![Созданные ключи](images/IAM/4.png)

**Шаг 3.** Добавьте эти ключи в секреты GitHub (см. раздел выше).

## Секреты для деплоя (id и ключ)

Вы написали код. Но что делать с чувствительными ключами, которые не должны попасть в чужие руки?

Правильно — создаём секреты. Для этого в облаке есть сервис **Lockbox**. Там можно хранить секреты, и о них никто не узнает.

### Как пользоваться

**Шаг 1.** Создаём Lockbox.

![Создание Lockbox](images/Lockbox/1.png)

**Шаг 2.** Создаём секрет.

Важно указать, что секрет пользовательский, и задать один или несколько ключей. Можно создать пустой секрет — при необходимости его удалим.

![Создание секрета](images/Lockbox/2.png)

**Шаг 3.** Использование секрета.

Секрет создан. Теперь нужно его использовать. В проекте должен быть файл, где мы читаем переменные из окружения (`.env`) и используем их. Пример — в `code/config.py`.

Как переменные попадают в окружение? В `ci.yml` (файл пайплайна) на строке 69 мы начинаем указывать наши секреты. Пример на строке 70:

```
secret environment-variable=BIG_SECRET,id=qwe123rty4567890,key=BIG_SECRET_LOCKBOX
```

### Разбор по порядку

| Параметр | Описание |
|----------|----------|
| `environment-variable=BIG_SECRET` | Имя переменной в окружении (как в `.env`) |
| `id=qwe123rty4567890` | ID секрета в Lockbox (скопировать из панели облака) |
| `key=BIG_SECRET` | Имя ключа внутри секрета в Lockbox |

> **Несколько ключей:** добавляем их в Lockbox, а в `.yml` — несколько строк, как выше. У каждой **не последней** строки в конце должно быть ` \` (пробел и обратный слэш), иначе не заработает.

## Dockerfile и всё, что с ним связано

Dockerfile — это инструкция, как собрать проект в контейнер. Он лежит в корне проекта.

> Если у вас несколько Dockerfile, называйте их, например: `one.Dockerfile`, `two.Dockerfile` и т.д.

### Разбор построчно

```
FROM python:3.12 AS base
```

Используем общедоступный образ Python 3.12 и называем его `base`. Похоже на объявление переменной.

```
WORKDIR /CI-CD_template
```

Указываем Docker рабочую директорию `CI-CD_template`.

```
RUN pip install --upgrade pip setuptools wheel
```

Обновляем pip, setuptools и wheel до последних версий.

```
FROM base AS flask-app
```

Второй этап сборки — образ с Flask-приложением. Называем его `flask-app`.

```
COPY . /CI-CD_template/
```

Копируем все файлы проекта в образ. Укажите название вашей папки с проектом.

```
ENV PYTHONPATH=/CI-CD_template
```

Добавляем путь к проекту в `PYTHONPATH`, чтобы Python находил модули.

```
ENV FLASK_APP=flask_app:app
```

Указываем Flask, какое приложение запускать: модуль `flask_app` и объект `app`. Если приложение в другом файле (например, `main.py`), укажите `main:app`.

```
WORKDIR /CI-CD_template
```

Рабочая директория — корень проекта.

```
RUN pip install --no-cache-dir -r requirements.txt
```

Устанавливаем зависимости из `requirements.txt`.

```
WORKDIR /CI-CD_template/code
```

Переходим в папку с кодом приложения.

```
CMD ["flask", "--app", "flask_app:app", "run", "--debug", "--host=0.0.0.0", "--port=8080"]
```

Запускаем Flask: `--app` — приложение, `--debug` — режим отладки, `--host=0.0.0.0` — слушаем все интерфейсы, `--port=8080` — порт (Yandex Cloud ожидает 8080).

### Что такое image name и target при сборке Docker-образа

**image name** — имя вашего Docker-образа, которое вы задаёте при сборке и публикуете в реестре. Обычно включает название проекта (например, `flask-app`). Используется в тегах и при развёртывании.

В пайплайне image name подставляется в команды сборки и деплоя:

```
cr.yandex/$CR_REGISTRY/$CR_REPOSITORY:<image name>-$IMAGE_TAG
```

**target** — имя этапа сборки (build stage) в `Dockerfile`. Используется при multi-stage build. В нашем случае этап называется `flask-app`:

```
FROM base AS flask-app
```

При сборке можно указать нужный этап:

```
docker build --target flask-app ...
```

Это позволяет собирать только нужную часть и не тянуть лишние зависимости.

### Зачем они нужны в yml-пайплайне

В `.github/workflows/ci.yml` значения `image name` и `target` подставляются в команды сборки и публикации образа.

**Примеры (см. yml):**

- `"--target [[README_Dockerfile_target]]"` — сюда подставляется имя этапа из Dockerfile (например, `flask-app`).
- `[[README_Dockerfile_image_name]]` — сюда подставляется название образа (например, `flask-app`) для тегирования и публикации.
